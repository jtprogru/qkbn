# ROADMAP — План развития qkbn

## Обзор

Этот документ описывает план развития утилиты **qkbn** — локального веб-сервера с Kanban-доской для визуализации сессий Qwen-code.

---

## v0.2.0 — Гибкая конфигурация запуска

### 1. Флаг для настройки порта (`--port`)

**Цель:** Позволить пользователю указывать порт веб-сервера через CLI-флаг.

**Требования:**
- Флаг `--port` (или `-p`) с типом `int`
- Значение по умолчанию: `9090`
- Валидация диапазона портов (1–65535)
- Вывод ошибки при недоступности порта

**Пример использования:**
```bash
qkbn --port 8080
qkbn -p 3000
```

**Критерии приемки:**
- [ ] Флаг парсится через `flag.IntVar`
- [ ] При невалидном порте выводится понятная ошибка
- [ ] Порт логгируется при старте сервера
- [ ] Покрыто unit-тестами (валидация, значение по умолчанию)

---

### 2. Флаг для настройки директории с задачами (`--todos-dir`)

**Цель:** Позволить указывать кастомную директорию с JSON-файлами задач.

**Требования:**
- Флаг `--todos-dir` (или `-d`) с типом `string`
- Значение по умолчанию: `~/.qwen/todos/`
- Поддержка `~` в пути (расширение до домашней директории)
- Проверка существования директории при старте
- Вывод ошибки если директория недоступна

**Пример использования:**
```bash
qkbn --todos-dir /custom/path/to/todos
qkbn -d ~/my-todos
```

**Критерии приемки:**
- [ ] Флаг парсится через `flag.StringVar`
- [ ] Функция `expandPath` обрабатывает `~`
- [ ] Проверка директории до запуска сервера
- [ ] Покрыто unit-тестами (расширение пути, несуществующая директория)

## v0.3.0 — Модульная архитектура UI

### 3. Вынос веб-интерфейса в отдельный пакет

**Цель:** Отделить логику HTTP-сервера и шаблонов от `main.go` для лучшей поддерживаемости.

**Требования:**
- Создать пакет `internal/http` или `pkg/http`
- Перенести `kanbanHandler` в отдельный файл
- Вынести HTML-шаблон в `templates/kanban.html`
- Использовать `embed.FS` для встраивания шаблона **при запуске** (не при сборке)
- Сохранить возможность сборки в один бинарник

**Структура после изменений:**
```
qkbn/
├── cmd/qkbn/
│   └── main.go          # Только CLI и инициализация
├── internal/http/
│   └── server.go        # HTTP-сервер и хендлеры
├── templates/
│   └── kanban.html      # HTML-шаблон доски
└── ...
```

**Важно:**
- Шаблон должен встраиваться в бинарник через `//go:embed`
- Это происходит **на этапе компиляции**, но используется **при запуске**
- Не требуется внешних файлов после сборки

**Критерии приемки:**
- [ ] `main.go` не содержит HTML-шаблонов
- [ ] Хендлер вынесен в отдельный пакет
- [ ] Шаблон используется через `embed.FS`
- [ ] Сборка работает без ошибок
- [ ] Покрыто интеграционными тестами (HTTP-запросы)

## v0.4.0 — Поддержка множественных сессий

### 4. Отображение всех сессий Qwen-code

**Цель:** Показывать все задачи из всех JSON-файлов в директории, с группировкой по сессиям/проектам.

**Проблема:**
Сейчас читается только **один** файл (последний изменённый). Пользователь может иметь несколько активных сессий Qwen-code, но видит только одну.

**Решение:**
- Читать **все** JSON-файлы из директории
- Группировать задачи по источнику (сессии/проекту)
- Отображать на доске с визуальным разделением

**Варианты реализации:**

#### Вариант A: Группировка по файлам (сессиям)
- Каждая сессия — отдельная "карточка" на доске
- Заголовок группы = имя файла или название проекта
- Переключение между группами через dropdown или вкладки

#### Вариант B: Сводная доска с метками
- Все задачи в общих колонках
- Цветовая маркировка по сессиям
- Фильтр по сессии (опционально)

#### Вариант C: Иерархическая доска
- Слева — список сессий
- Справа — Kanban для выбранной сессии
- Опция "показать все" для сводного вида

**Требования:**
- Чтение всех `*.json` файлов из директории
- Парсинг каждого файла в общую структуру
- Группировка по полю `sessionName` или имени файла
- UI с визуальным разделением групп
- Сортировка: активные сессии сверху

**Пример структуры данных:**
```go
type Session struct {
    Name   string
    Tasks  []Task
    Updated time.Time
}

type Board struct {
    Sessions []Session
    Todo     []Task // aggregated
    InProgress []Task
    Done     []Task
}
```

**Критерии приемки:**
- [ ] Читаются все JSON-файлы из директории
- [ ] Задачи группируются по сессиям
- [ ] UI отображает название сессии для каждой задачи
- [ ] Добавлена сортировка сессий (по времени обновления)
- [ ] Покрыто unit-тестами (чтение нескольких файлов, пустая директория)
- [ ] Интеграционные тесты для множественных сессий

## Технические заметки

### Embed vs Runtime
- `//go:embed` работает **на этапе компиляции**
- Файлы встраиваются в бинарник
- При запуске не требуется доступ к файловой системе
- Это соответствует требованию "цеплять при запуске"

### Обратная совместимость
- Флаги должны быть опциональными
- Значения по умолчанию сохраняют текущее поведение
- Старые команды должны работать без изменений

### Тестирование
- Unit-тесты: парсинг флагов, валидация, группировка задач
- Интеграционные: HTTP-запросы, рендеринг шаблона
- Coverage цель: >60%
